# 从需求到成熟设计

一份设计方法论，用本仓库的 `tensor.hpp` 作为贯穿全文的例子。目的不是解释代码
*做了什么*——头部注释已经讲得很清楚了——而是解释这样一份设计是*怎么被推导出来*的。

## 核心认知：设计是自底向上盖起来的，不是从上往下写出来的

新手容易以为高手是打开一个空文件，从第一行一路敲到最后一行。高性能代码不是这么写的，
它是一层一层盖起来的：

1. **数据布局** —— 字节在内存里怎么摆。
2. **访问模式** —— 数据怎么被读、怎么被写。
3. **并发模型** —— 谁、在什么时候，被允许碰它。
4. **算法** —— 真正的运算，最后才填进去。

每一层只回答一个问题：*在下面各层的决定已经定死的前提下，这一层被逼着必须怎么做？*
语言特性（模板、自定义分配器、`shared_mutex`、SIMD intrinsics）从来不是起点。
它们是**最后一步**——是前面的决定逼出来的形状。

一个有用的推论：如果你有硬件直觉、却觉得自己"高级 C++ 不行"，其实你并不落后。好的
高性能设计恰恰是*从硬件往上推*的。一旦底下几层定死，语言特性就成了唯一可能的答案——
于是你自然会伸手去用它，而不是抽象地干啃。

## 从 `tensor.hpp` 倒推出的设计链

下面每一步都是一个决定，而每个决定都被上一个逼出来。请按顺序读——重点是这条*链*，
不是孤立的某个选择。

### 决策 0 —— 数据在内存里怎么摆？

这是地基，先定死。硬件事实很简单：连续内存 + 顺序访问 = cache 保持命中 = 快。

所以存储是一块**一维连续 buffer**，二维形状只是在它之上的一次索引换算：

```cpp
T& operator()(std::size_t r, std::size_t c) { return storage_[r * cols_ + c]; }
```

这是整个设计里最重要的一个决定，先定死它。用 `vector<vector<T>>` 会把每一行散落在
内存的不同地方，不停地 cache miss。**布局错了，上面再怎么优化都白搭。** 把这个决定
做对，比后面所有 SIMD kernel 加起来还值钱。

### 决策 1 —— 想上 SIMD，对齐就自动被逼出来了

AVX-512 的 load 指令喜欢 64 字节对齐，而 `std::vector` 不保证对齐。我们又不想放弃
`std::vector`（要它的 RAII 和 `resize`），所以只替换掉控制内存的那一部分：分配器。
这就是 `AlignedAllocator` 存在的原因。

注意这里的模板*为什么*出现。`std::vector<T, Allocator>` 要求提供一个分配器，而这个
分配器本身就是个类模板。这个模板是**被接口逼出来的**，不是为了用而用。这是一个通用
规律：某个特性冒出来，是因为它是唯一能塞进上一层约束的形状——而不是因为有人想用它。

### 决策 2 —— 多个线程写同一块 buffer，怎么才不打架？

一旦切换到硬件视角，这就是纯逻辑推理：

- 事实：两个线程写**同一条 cache line** 上的不同元素，硬件会强制同步这条线
  （false sharing）——极慢。
- 结论：让每个线程负责的区间边界，**落在 cache line 边界上**。一条 cache line 装
  16 个 float，所以切块大小向上取整到 16 的整数倍。

这一个事实，就是 `parallel_chunks` 里这一行的全部动机：

```cpp
chunk = ((chunk + granularity - 1) / granularity) * granularity; // 向上取整到整数条 cache line
```

多线程之所以让人觉得"难"，主要是因为你不清楚*到底要防什么*。一旦从硬件角度把威胁
点名清楚（防 false sharing、防 data race），代码写法几乎是被逼出来的。

### 决策 3 —— 读写锁，因为读远多于写

前向传播时，绝大多数操作是读（读形状、读数据、算元素），真正改变结构的
（`resize`）很少。用普通 `mutex` 会让并发的读也毫无必要地互相排队。所以用
`shared_mutex`：读操作拿 `shared_lock`（可并发），结构性写操作拿 `unique_lock`
（独占）。

```cpp
std::size_t rows() const { std::shared_lock lock(mutex_); return rows_; } // 读，可并发
void resize(/*...*/)     { std::unique_lock lock(mutex_); /* ... */ }     // 写，独占
```

同样：先有需求（读占主导），再有选型（`shared_mutex`）。不是"我知道 `shared_mutex`，
所以用它"。

### 决策 4 —— 运行时选 SIMD，只探测一次

不能假设用户的 CPU 有 AVX-512。所以：编译时把三条路径（AVX-512 / AVX2/ 标量）都
编出来，运行时用 CPUID 探测一次（还要加 XGETBV，因为 AVX-512 还需要操作系统支持
更宽的寄存器状态），把选中的函数塞进一张函数指针表。之后热路径上就没有分支了：

```cpp
struct F32Kernels { BinaryF32Fn add; BinaryF32Fn mul; UnaryF32Fn relu; };
```

"探测一次、缓存、之后热路径永不再分支"是高性能库的标准套路。

## 这套思考的形状

把整条链串起来，会浮现出一个固定的思考顺序：

1. **先问硬件要什么** —— 连续内存、对齐、避免 false sharing。
2. **再问需求隐含了什么访问模式** —— 读占主导 ⇒ 读写锁；大块逐元素 buffer ⇒
   SIMD + 分块。
3. **最后才让 C++ 特性来服务上面的决定** —— 要对齐 ⇒ 自定义分配器；要分块 ⇒
   模板化的 `parallel_chunks`；要类型无关 ⇒ `template <typename T>`。

**特性永远是最后一步，是被上面的决定逼成那个形状的。**"风格单一、不用模板"通常
不是能力的天花板——只是你还没走到一个"这些特性是唯一解"的问题面前。带着具体问题去
学，一次学一个，才记得住。

## 分层看：这里的"2D"到底指什么

关于这份代码的一个敏锐观察：SIMD kernel 是在"两个向量之间"运算的，那是不是意味着它
只能处理 2D 张量？答案和直觉正好相反——**正因为** kernel 是向量对向量的，它才是
**与维度无关**的。

- **计算层 —— 对维度漠不关心。** kernel 的签名是
  `add_f32_avx2(const float* x, const float* y, float* out, std::size_t n)`，
  它只知道两个指针、一个输出、一个元素总数 `n`。逐元素加法对 `[a,b,c,d]` 和
  `[[a,b],[c,d]]` 给出完全一样的结果——拍平了加就行。`add` / `mul` / `relu` 现在
  就已经是 N 维通用的；扩到 3D、4D 一行都不用改。

- **元数据层 —— 这里才把 2D 写死。** 只有两个形状字段（`rows_`、`cols_`），
  `shape()` 返回一个 `pair`，`operator()` 只收 `(r, c)`。"2D"是这个索引约定造成
  的，不是计算造成的。把它们换成 `std::vector<std::size_t> shape_` 加一个变参的
  `operator()`，那块一维 buffer 和所有 SIMD kernel 全部原样复用。

实用结论：逐元素操作可以免费拍平，但真正*需要*维度结构的操作——矩阵乘和卷积——不是
逐元素的，而且这里还没实现。维度真正开始有意义的地方是它们，不是 SIMD kernel。

## 怎么练

1. **先单线程重写一遍这个 tensor**，别看并行部分。目标只有一个：布局对（一维、连续、
   对齐）+ 标量循环写对。一个布局正确的单线程版本，往往就比朴素的第一版快好几倍——
   先把地基打对，再谈并行。
2. **然后只学一个概念：false sharing。** 配上硬件直觉一点就通，通了之后
   `parallel_chunks` 你自己就能写出来。
3. **别硬学模板。** 等哪天你想让一个 buffer 类型既能装 `float` 又能装 `int`，
   `template <typename T>` 自然就成了显而易见的形状。带着问题学，一次一个。
